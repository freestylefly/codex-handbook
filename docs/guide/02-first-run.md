# 第一次让 Codex 改代码

第一次实战不要选“重构整个项目”。选一个小、可验证、失败也容易回滚的任务。

## 推荐任务

适合新手的任务：

- 修复一个文案错别字。
- 给一个纯函数补测试。
- 更新 README 里的过期命令。
- 解释一个小模块，并补充必要注释。
- 修复一个已经有失败测试覆盖的 bug。

不适合第一次尝试的任务：

- 大规模架构重构。
- 跨多个服务的迁移。
- 没有测试的核心业务逻辑改动。
- 涉及生产凭据、账单、权限和删除数据的操作。

## 一个可直接复制的任务说明

```text
请修复当前仓库中最小范围的一个测试失败。

要求：
1. 先运行测试，确认失败信息。
2. 阅读相关代码和测试，不要做无关重构。
3. 修改最少必要文件。
4. 修复后重新运行相关测试。
5. 最后总结：失败原因、改了哪些文件、验证命令和剩余风险。
```

## 观察 Codex 的过程

重点看四件事：

- 它有没有先读上下文。
- 它有没有控制改动范围。
- 它有没有运行验证命令。
- 它有没有诚实说明没能验证的部分。

## 你可以如何接住结果

完成后请自己再看一遍 diff：

```bash
git diff
```

如果结果满意，再提交：

```bash
git add .
git commit -m "fix: resolve failing test"
```

Codex 可以极大加速你，但最终合并代码的人仍然要对代码负责。
