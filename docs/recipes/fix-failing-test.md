# 修复一个测试失败

这是最适合新手的 Codex 实战任务之一，因为它有明确的失败信号和验证闭环。

## 适用场景

- 仓库已经有测试。
- 当前有一个或少量测试失败。
- 失败和环境无关，或环境问题也能被清楚识别。

## 任务说明

```text
请修复当前仓库中的测试失败。

要求：
1. 先运行测试命令，记录失败信息。
2. 根据失败信息阅读相关测试和实现代码。
3. 找到根因后再修改，不要做无关重构。
4. 修改最少必要文件。
5. 修复后重新运行失败相关的测试。
6. 最后总结根因、改动文件、验证命令和剩余风险。
```

## 你要重点检查什么

- Codex 是否真的复现了失败。
- 它是否理解测试期望，而不是简单改测试迁就实现。
- 修复是否影响其他行为。
- 是否新增了必要的边界测试。

## 常见追问

如果它只改测试：

```text
请解释为什么应该修改测试而不是实现。这个测试原本表达的业务期望是什么？有没有官方需求、README 或其他测试可以佐证？
```

如果它跑不通测试：

```text
请区分当前失败是环境问题、依赖问题还是代码问题。不要继续改业务代码，先给出阻塞原因和可执行的修复建议。
```
